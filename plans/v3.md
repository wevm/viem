# Viem v3 Plan

## Target Shape

```ts
import { Account, BaseError, Chain, Client, Transport, http } from 'viem'
import { mainnet } from 'viem/chains'
import { Hex, Value } from 'viem/utils'
import * as actions from 'viem/actions'

const client = Client.create({
  chain: mainnet,
  transport: http(),
})
  .extend(actions.public())
  .extend(actions.contract())
  .extend(actions.ens())

const blockNumber = await client.public.getBlockNumber()
const balance = await actions.public.getBalance(client, { address })
```

## Principles

- Viem v3 is a rewrite into the new Ox/Zile-inspired structure, not a mechanical in-place refactor of v2 modules.
- Root `viem` exports PascalCase viem-owned modules, the special `BaseError`, and flat transport factories.
- Core module subpaths are PascalCase: `viem/Account`, `viem/Chain`, `viem/Client`, and `viem/Transport`.
- Actions are only exposed through lowercase namespace imports: `import * as actions from 'viem/actions'`.
- Actions are nested by capability: `public`, `wallet`, `contract`, `ens`, and `test`.
- Action namespaces are callable decorator factories with standalone action properties: `actions.public()` and `actions.public.getBalance(...)`.
- Decorated clients expose nested capability methods: `client.public.getBalance`, `client.contract.read`, `client.ens.getAddress`.
- Chains stay flat under `viem/chains`; chain helpers move to `Chain.*`.
- Utilities live under `viem/utils` and `viem/utils/<Module>`, proxying Ox module APIs exactly where Ox has coverage.
- Tempo mirrors root viem with lowercase `actions` subpaths and PascalCase domain modules/subpaths.
- OP Stack, ZKsync, Celo, and Linea extension entrypoints are removed in v3.
- Extension removal does not remove their chain constants from `viem/chains`.
- The `viem/experimental` root entrypoint is removed in v3.
- ERC-7739, ERC-7715, ERC-7811, ERC-7821, and ERC-7895 experimental modules are removed.
- ERC-7846 graduates out of experimental into `actions.wallet` and `client.wallet` after an API review.
- Retained behavior must be checked against v2 after each module rewrite and before public cutover.
- No runtime compatibility shims.
- Every migrated public module gets an API review before code migration starts.

## Rewrite Strategy

V3 should be implemented as a new source tree in the package. The legacy v2
implementation lives in `src-old` for reference and behavior comparison only;
the active package source is `src`.

Avoid gradually reshaping v2 files into v3 files. Instead, build clean v3
modules, switch public entrypoints to those modules, then delete the replaced
v2 implementation when it is no longer useful as reference.

The rewrite proceeds from dependency leaves to roots: internal primitives and
utilities first, then RPC/domain formatters, then chains/accounts, then
transports, clients, actions, and public entrypoints.

For each rewritten domain:

1. Capture the reviewed v3 API shape in this plan or the migration skill.
2. Implement the v3 module in the new structure.
3. Compare retained behavior against the v2 implementation in the module's normal runtime and type tests.
4. Switch the public entrypoint to the v3 module.
5. Delete the old implementation and removed v2 APIs.
6. Update migration guidance for every public break.

The v2 comparison should be local to the module being rewritten. While the old
and new implementations coexist, tests can import both directly and assert the
same retained runtime behavior, RPC shapes, formatting, signing, and inference.
Intentional v3 breaks should be recorded in the module review instead of hidden
behind broad compatibility tests.

## Rewrite Review Gate

Before rewriting any public module, review and record:

- Canonical imports and subpaths.
- Exported runtime values.
- Exported types and namespace-attached types.
- Method names after removing redundant domain words.
- Error placement and names.
- RPC boundary behavior, including accepted input quantities and normalized output values.
- Retained v2 behaviors to compare after the rewrite.
- Removed v2 APIs, aliases, deprecated exports, and migration guidance.

Do not start a module rewrite until the reviewed API shape is captured in this
plan or the migration skill.

## Reviewed API: `Hex`, `Bytes`, And `Value` Utilities

Canonical imports:

```ts
import { Bytes, Hex, Value } from 'viem/utils'
import * as Hex from 'viem/utils/Hex'
```

Runtime exports:

- `Hex`, `Bytes`, and `Value` proxy Ox modules exactly.
- Deep utility subpaths mirror Ox module names and method names.
- V2 flat wrappers such as `toHex`, `hexToBytes`, `formatUnits`, and
  `parseEther` are removed during public cutover.

Retained behavior to compare:

- Hex/bytes encoding and decoding for booleans, numbers, strings, and byte
  arrays.
- Hex/bytes concatenation, padding, slicing, and trimming.
- Value formatting and parsing for arbitrary decimals, ether, and gwei.
