# Viem v3 Plan

## Target Shape

```ts
import { Account, BaseError, Chain, Client, Transport, http } from 'viem'
import { mainnet } from 'viem/chains'
import { Hex, Value } from 'viem/utils'
import * as actions from 'viem/actions'

const client = Client.create({
  chain: mainnet,
  transport: http(),
})
  .extend(actions.public())
  .extend(actions.contract())
  .extend(actions.ens())

const blockNumber = await client.public.getBlockNumber()
const balance = await actions.public.getBalance(client, { address })
```

## Principles

- Root `viem` exports PascalCase viem-owned modules, the special `BaseError`, and flat transport factories.
- Core module subpaths are PascalCase: `viem/Account`, `viem/Chain`, `viem/Client`, and `viem/Transport`.
- Actions are only exposed through lowercase namespace imports: `import * as actions from 'viem/actions'`.
- Actions are nested by capability: `public`, `wallet`, `contract`, `ens`, and `test`.
- Action namespaces are callable decorator factories with standalone action properties: `actions.public()` and `actions.public.getBalance(...)`.
- Decorated clients expose nested capability methods: `client.public.getBalance`, `client.contract.read`, `client.ens.getAddress`.
- Chains stay flat under `viem/chains`; chain helpers move to `Chain.*`.
- Utilities live under `viem/utils` and `viem/utils/<Module>`, proxying Ox module APIs exactly where Ox has coverage.
- Tempo mirrors root viem with lowercase `actions` subpaths and PascalCase domain modules/subpaths.
- OP Stack, ZKsync, Celo, and Linea extension entrypoints are removed in v3.
- Extension removal does not remove their chain constants from `viem/chains`.
- The `viem/experimental` root entrypoint is removed in v3.
- ERC-7739, ERC-7715, ERC-7811, ERC-7821, and ERC-7895 experimental modules are removed.
- ERC-7846 graduates out of experimental into `actions.wallet` and `client.wallet` after an API review.
- No runtime compatibility shims.
- Every migrated public module gets an API review before code migration starts.

## Module Review Gate

Before migrating any public module, review and record:

- Canonical imports and subpaths.
- Exported runtime values.
- Exported types and namespace-attached types.
- Method names after removing redundant domain words.
- Error placement and names.
- RPC boundary behavior, including accepted input quantities and normalized output values.
- Removed v2 APIs, aliases, deprecated exports, and migration guidance.

Do not start a module migration until the reviewed API shape is captured in this plan or the migration skill.

## Phase 2: Core Modules

- [ ] Review the `Account` API before migration.
- [ ] Add `Account` module with explicit constructors: `fromPrivateKey`, `fromMnemonic`, `fromHdKey`, `fromJsonRpc`, `fromLocal`, `random`, and `from`.
- [ ] Review the `Chain` API before migration.
- [ ] Add `Chain` module with `Chain.define`; accepted quantity inputs may be hex, number, or bigint, while normalized chain IDs are bigint.
- [ ] Review the `Client` API before migration.
- [ ] Add `Client` module with `Client.create` and nested `.extend(...)` behavior.
- [ ] Review the `Transport` API before migration.
- [ ] Add `Transport` module for transport types and low-level helpers while keeping `http`, `webSocket`, `custom`, and `fallback` flat.
- [ ] Review the `BaseError` API before migration.
- [ ] Add viem `BaseError` built on Ox `BaseError` with viem version metadata.
- [ ] Keep key handling internal and use Ox crypto modules directly.

## Phase 3: Action Namespaces

- [ ] Review the `actions.public` API before migration.
- [ ] Convert public client actions to `actions.public.*` and `client.public.*`.
- [ ] Review the `actions.wallet` API before migration.
- [ ] Convert wallet actions to `actions.wallet.*` and `client.wallet.*`.
- [ ] Review the `actions.contract` API before migration.
- [ ] Convert contract actions to `actions.contract.*` and `client.contract.*`.
- [ ] Review the `actions.ens` API before migration.
- [ ] Convert ENS actions to `actions.ens.*` and `client.ens.*`, dropping redundant `Ens` method infixes.
- [ ] Review the `actions.test` API before migration.
- [ ] Convert test actions to `actions.test.*` and `client.test.*`.
- [ ] Use `declare namespace` attached to action functions for `Options`, `ReturnType`, and `ErrorType`.
- [ ] Delete `GetXParameters`, `GetXReturnType`, and `GetXErrorType` public aliases as actions migrate.

## Phase 4: Ox Utility Migration

- [ ] Review each Ox-backed utility module API before migration.
- [ ] Make `viem/utils` an Ox-exact module proxy entrypoint.
- [ ] Add deep utility module subpaths such as `viem/utils/Hex`, `viem/utils/Address`, and `viem/utils/TransactionRequest`.
- [ ] Remove flat utility wrappers and old lowercase utility subpaths.
- [ ] Remove flat `abitype` re-exports from `viem/utils`.
- [ ] Review each viem-specific utility module API before migration.
- [ ] Keep viem-specific public utilities in module form, such as `StateOverride`, `NonceManager`, and `Ccip`.

## Phase 5: RPC And Formatters

- [ ] Review each RPC/formatter domain module API before migration.
- [ ] Replace public `formatX` helpers with module methods such as `Block.fromRpc`, `Transaction.fromRpc`, and `TransactionRequest.toRpc`.
- [ ] Keep chain property name `formatters`.
- [ ] Change formatter entries from `format` to `fromRpc` and `toRpc`.
- [ ] Normalize incoming RPC values to viem domain objects and serialize outgoing quantities to RPC hex.
- [ ] Update remaining formatter overrides, including Tempo, to express only chain-specific deltas.

## Phase 6: Extension Removal And Tempo

- [ ] Remove `viem/op-stack`, `viem/zksync`, `viem/celo`, and `viem/linea` public entrypoints.
- [ ] Remove OP Stack, ZKsync, Celo, and Linea package export-map entries, docs, and tests.
- [ ] Remove OP Stack, ZKsync, Celo, and Linea source modules unless still needed internally by chains or tests.
- [ ] Record removed OP Stack, ZKsync, Celo, and Linea entrypoints in the migration skill.
- [ ] Review the Tempo extension API before migration.
- [ ] Normalize Tempo to lowercase `actions` subpaths and PascalCase domain modules/subpaths.

- Type tests for root modules, action namespace imports, nested client decorators, and utility modules.
- Ox parity smoke tests for selected utility modules.
- Formatter tests for default chains plus remaining formatter overrides such as Tempo.
- Targeted package checks during each phase.
- Final checks: `pnpm check:types`, `pnpm build`, and `pnpm test:build`.
